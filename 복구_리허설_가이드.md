# 새롬 GLS 데이터베이스 복구 리허설 가이드

> 최종 수정: 2026-04-13  
> 작성: Phase 1 보안 강화 작업  
> 리허설 주기: 분기 1회 (1월, 4월, 7월, 10월)

---

## 1. 개요

이 문서는 Neon PostgreSQL 데이터베이스 장애 발생 시 복구 절차와 분기별 복구 리허설 방법을 정의합니다.
실제 장애 상황에서 당황하지 않도록 정기적으로 리허설을 수행해야 합니다.

---

## 2. 복구 시나리오

| 시나리오 | 원인 | 복구 방법 | 예상 소요 시간 |
|---------|------|----------|-------------|
| 데이터 오염 | 잘못된 배치 작업/SQL 실행 | Neon 시점 복원 (Point-in-Time Recovery) | 5~10분 |
| 전체 DB 손실 | Neon 장애/삭제 | GitHub Actions 일간 백업에서 복원 | 15~30분 |
| 특정 테이블 복원 | 실수로 대량 삭제 | Neon 브랜치 생성 후 부분 복원 | 10~20분 |
| 스키마 마이그레이션 롤백 | 마이그레이션 오류 | Prisma migrate 롤백 + 데이터 복원 | 20~40분 |

## 3. Neon 시점 복원 (PITR) 절차

Neon은 자동으로 WAL(Write-Ahead Log)을 저장하므로 특정 시점으로 복원할 수 있습니다.

### 3-1. Neon 콘솔에서 복원

1. [Neon Console](https://console.neon.tech) 접속
2. 프로젝트 선택 (saeromgls)
3. 좌측 메뉴 "Branches" 클릭
4. "Create Branch" 클릭
5. "From" 옵션에서 복원 시점 선택:
   - "Point in time" 선택
   - 복원할 날짜/시간 입력 (UTC 기준, KST = UTC + 9)
6. 브랜치 이름 입력 (예: `restore-20260413`)
7. "Create Branch" 실행

### 3-2. 복원 브랜치에서 데이터 확인

```bash
# 복원 브랜치의 connection string으로 접속
psql "postgresql://[user]:[password]@[restore-branch-host]/neondb?sslmode=require"

# 데이터 무결성 확인 쿼리
SELECT COUNT(*) FROM workers WHERE status = 'ACTIVE';
SELECT COUNT(*) FROM work_items WHERE started_at > NOW() - INTERVAL '7 days';
SELECT COUNT(*) FROM subscriptions;
```

### 3-3. 운영 브랜치에 반영

**방법 A: 전체 교체** (전체 DB 손상 시)
1. Neon 콘솔에서 운영 브랜치(main)의 엔드포인트를 복원 브랜치로 변경
2. Vercel 환경변수 `DATABASE_URL`을 새 connection string으로 갱신
3. API 서버 재배포

**방법 B: 부분 복원** (특정 테이블만 복원)
```bash
# 복원 브랜치에서 데이터 추출
pg_dump -h [restore-host] -U [user] -d neondb -t workers --data-only > workers_restore.sql

# 운영 브랜치에 적용
psql -h [main-host] -U [user] -d neondb < workers_restore.sql
```

## 4. GitHub Actions 백업에서 복원

일간 백업은 GitHub Actions를 통해 자동 실행되며, 아티팩트로 저장됩니다.

### 4-1. 백업 파일 다운로드

1. GitHub 리포지토리 > Actions 탭
2. "DB Backup" 워크플로우 선택
3. 복원할 날짜의 실행 결과 선택
4. Artifacts에서 백업 파일 다운로드

### 4-2. 복원 실행

```bash
# 백업 파일 압축 해제
gunzip backup_20260413.sql.gz

# Neon 새 브랜치 생성 (빈 상태)
# Neon Console > Branches > Create Branch > "Empty" 선택

# 백업 데이터 복원
psql "postgresql://[user]:[password]@[new-branch-host]/neondb?sslmode=require" < backup_20260413.sql

# 마이그레이션 상태 확인
npx prisma migrate status
```

## 5. 복원 후 검증 체크리스트

복원 완료 후 반드시 아래 항목을 순서대로 검증합니다.

### 5-1. 데이터 무결성 확인

- [ ] workers 테이블: 활성 작업자 수 확인 (현재 약 19명)
- [ ] work_items 테이블: 최근 7일 작업 기록 존재 확인
- [ ] subscriptions 테이블: 활성 구독 상태 확인
- [ ] sites 테이블: 사업장 정보 확인
- [ ] classifications 테이블: 분류 코드 존재 확인

### 5-2. 관계 무결성 확인

```sql
-- 고아 레코드 확인 (site 없는 worker)
SELECT COUNT(*) FROM workers w 
LEFT JOIN sites s ON w.site_id = s.id 
WHERE w.site_id IS NOT NULL AND s.id IS NULL;

-- 고아 작업 기록 (worker 없는 work_item)
SELECT COUNT(*) FROM work_items wi 
LEFT JOIN workers w ON wi.started_by_worker_id = w.id 
WHERE w.id IS NULL;
```

### 5-3. API 기능 확인

- [ ] 웹 대시보드 로그인 (ADM001 / PIN)
- [ ] 작업자 목록 조회 (/api/admin/workers)
- [ ] 오늘 작업 현황 조회 (/api/admin/work-items)
- [ ] 모바일 API 작업자 목록 (/api/mobile/workers)
- [ ] 구독 상태 조회 (/api/admin/subscriptions)

### 5-4. 외부 서비스 연결 확인

- [ ] Vercel API 배포 상태 정상
- [ ] Vercel Web 배포 상태 정상
- [ ] Neon DB 연결 정상
- [ ] Sentry 에러 모니터링 정상

## 8. 분기별 리허설 절차

### 리허설 일정

- 1분기: 1월 둘째 주 월요일
- 2분기: 4월 둘째 주 월요일
- 3분기: 7월 둘째 주 월요일
- 4분기: 10월 둘째 주 월요일

### 리허설 순서

1. **사전 준비** (5분)
   - 현재 운영 DB 상태 스냅샷 기록 (작업자 수, 최근 작업 수 등)
   - 리허설용 Neon 브랜치 생성 예정 시점 기록

2. **시점 복원 테스트** (15분)
   - Neon PITR로 30분 전 시점 브랜치 생성
   - 데이터 무결성 체크리스트 실행 (5-1, 5-2)
   - 복원 브랜치 연결 테스트 (psql 접속 확인)

3. **백업 파일 복원 테스트** (20분)
   - 최근 GitHub Actions 백업 아티팩트 다운로드
   - 별도 Neon 브랜치에 복원
   - 데이터 비교 (운영 vs 복원)

4. **API 기능 테스트** (10분)
   - 복원된 DB로 로컬 API 서버 기동
   - 체크리스트 5-3 실행

5. **정리 및 기록** (10분)
   - 리허설용 브랜치 삭제
   - 결과 기록 (소요 시간, 발견된 문제, 개선 사항)
   - 다음 리허설 일정 확인

## 9. 주의사항

1. **운영 DB에서 직접 복원 작업 금지** - 반드시 별도 브랜치에서 먼저 검증
2. **복원 전 현재 상태 백업** - 복원 작업이 실패할 경우를 대비
3. **Vercel 환경변수 변경 시 주의** - DATABASE_URL 변경 시 모든 서비스에 영향
4. **UTC/KST 시간대 혼동 주의** - Neon PITR은 UTC 기준, 한국은 UTC+9
5. **암호화된 PII 데이터** - 복원 후 PII_ENCRYPTION_KEY가 동일한지 확인 (Phase 1: phone 암호화)
